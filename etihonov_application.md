**[ НАЗВАНИЕ ПРОЕКТА ]**

“Fathom”

**[ ПРОБЛЕМНОЕ ПОЛЕ ]**
Преподавателям технических и естественно-научных дисциплин (физика, математика, инженерия, программирование) для быстрой проверки знаний на занятиях, кружках или курсах необходим простой и специализированный инструмент. Существующие решения создают непреодолимый барьер:

* Moodle и аналогичные LMS - это как использовать рапиру для [операции на виноградинке](https://ic.pics.livejournal.com/dombusin/18706815/583853/583853_original.jpg). Они представляют собой сложные системы управления всем учебным процессом **вуза**. Их развертывание, администрирование и освоение непропорционально сложны для задачи преподавателя-предметника, который хочет просто и быстро создать специализированный тест с формулами и кодом. Также функционалы *курсов, чата и т.п.* просто не нужны.

* Конструкторы тестов (Google Forms, Quizlet) - слишком примитивны. Они не поддерживают необходимый для технических дисциплин контент: математические формулы (LaTeX), фрагменты кода с подсветкой синтаксиса, создание заданий на соответствие или упорядочивание.

Итоговая проблема: Преподаватель, ведущий дополнительное занятие или курс, оказывается в вакууме между *избыточно сложными корпоративными системами* и *слишком простыми потребительскими сервисами*. В результате он тратит время на преодоление ограничений инструментов, а не на создание качественных заданий.

**[ ЗАКАЗЧИК / ПОТЕНЦИАЛЬНАЯ АУДИТОРИЯ ]**

Основная аудитория: Преподаватель-энтузиаст (в вузе, колледже, на курсах, репетитор), который ведёт занятия по техническим дисциплинам на своём содержании и часто своими силами. Он ценит:

* **Простота и скорость**: возможность развернуть систему за 5 минут на своём ноутбуке или недорогом VPS. Также стоит отметить, что количество библиотек должно быть сокращено. Батарейки не нужны и устанавливать модули не нужно.

* **Специализация**: встроенная и бесшовная поддержка LaTeX для формул и подсветки кода «из коробки».

* **Контроль и конфиденциальность**: полный контроль над своими данными (все задания и результаты хранятся в его экосистеме).

Вторичная аудитория:

* Студенты/ученики: для них важна возможность быстро присоединиться к тесту по ссылке без сложной регистрации и получить ясную обратную связь.

* Методисты и авторы: заинтересованы в открытом формате хранения заданий, который позволяет вести версионность (например, через Git) и делиться банками вопросов.

**[ АППАРАТНЫЕ / ПРОГРАММНЫЕ ТРЕБОВАНИЯ ]** 

Продукт разрабатывается как кроссплатформенное веб-приложение, развертываемое одним из двух способов:

1. Режим «Персональный компьютер» (основной сценарий): Запуск на ноутбуке или ПК преподавателя для проведения занятий в классе или онлайн. Работает в локальной сети.

    * Клиент: Любой современный браузер (Chrome, Firefox, Edge последних версий).

    * Сервер: Docker Desktop или прямой запуск бинарного файла (для Windows/macOS/Linux). Минимальные требования: 1 CPU, 1 ГБ ОЗУ, 10 ГБ на диске.

2. Режим «Сервер»: Развертывание на публичном сервере(VDS) для постоянного доступа.

   * Сервер: Любой VDS с ОС Linux (Ubuntu 22.04 LTS или аналоги) и Docker. Минимальная конфигурация: 1 vCPU, 1 ГБ ОЗУ, 10 ГБ SSD.
    
**[ ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ ]**
1. Для преподавателя (создание и запуск):

    * Редактор из коробки: Единое поле ввода для условия вопроса с мгновенным предпросмотром. Поддержка смешанного контента: обычный текст, Markdown(для форматирования), LaTeX(формулы отображаются рядом), код(с автоматической подсветкой для популярных ЯП).

    * Типы вопросов, необходимые для технических предметов:

        * Один/несколько правильных ответов.

        * Свободный ответ (с проверкой по шаблону/регулярному выражению).

        * Установление соответствия.

        * Упорядочивание последовательности.

        * Числовой ответ (с указанием погрешности).

    * Работа с заданиями: Создание банка заданий, объединение в тесты/контесты.

    * Запуск теста: Получение короткой ссылки и QR-кода для доступа учеников. Настройка времени, количества попыток.

    * Анализ на лету: Мгновенное получение результатов после завершения теста: таблица ответов, статистика по вопросам, возможность выгрузки в CSV.

1. Для студента (прохождение):

    * Доступ по ссылке или QR-коду.

    * Чистый, адаптивный интерфейс для прохождения теста.

    * Автосохранение ответов.

    * Просмотр результата (если разрешил преподаватель): свои ответы и правильные.

**[ ПОХОЖИЕ / АНАЛОГИЧНЫЕ ПРОДУКТЫ ]**

1. Moodle Quiz: Институциональная LMS для управления учебным процессом всего вуза. Чрезвычайно сложен в развертывании(конечно, если не использовать docker), администрировании и ежедневном использовании для создания одного теста. Некоторые функции требуют отдельной настройки.
2. Google/Microsoft Forms: Универсальный облачный конструктор опросов. Не поддерживают LaTeX и код. Нет полного и прозрачного контроля над данными (они в облаке).
3. Stepik: Платформа для создания публичных курсов, особенно с уклоном в программирование. Ориентирован на публикацию курсов для широкой аудитории, менее удобен для приватного, быстрого тестирования своей группы.
4. iSpring Suite: Проприетарный авторский инструмент для корпоративного e-learning. Закрытый, платный, нельзя развернуть у себя.

Ключевое отличие: Этот проект - персональный open-source инструмент, который не пытается быть большой и хорошей во всём LMS, а решает одну задачу идеально: дать преподавателю-предметнику максимально простой и мощный способ создать и провести специализированный тест, сохранив полный контроль над данными и процессом.

**[ ИНСТРУМЕНТЫ РАЗРАБОТКИ, ИНФОРМАЦИЯ О БД ]**

* Бэкенд: Golang + Gin + Bun.

* Фронтенд: React + TypeScript + Vite. Формулы: KaTeX. Редактор кода: CodeMirror.

* БД: SQLite для режима «Персональный компьютер» и PostgreSQL для серверного режима.

* Развертывание: Docker/Docker Compose как основной способ; предоставление бинарных файлов для основных ОС.

**[ ЭТАПЫ РАЗРАБОТКИ ]**

* Проектирование модели данных и API (2 недели): Определение структуры для хранения вопросов разных типов, тестов, результатов.

* Разработка ядра API и базовых моделей (3 недели): Реализация на Gin, настройка аутентификации и ролей.

* Создание редактора вопросов (4 недели): Самый сложный этап. Разработка веб-интерфейса с переключением типов вопросов, полем для формул и вставки кода.

* Реализация механизма прохождения теста (2 недели): Интерфейс студента, логика таймера, отправки ответов.

* Разработка модуля аналитики и отчетов (2 недели): Страницы со статистикой, выгрузка данных.

* Тестирование, документация, подготовка к деплою (3 недели): Написание инструкции по развертыванию через Docker Compose, исправление ошибок.


**[ ВОЗМОЖНЫЕ РИСКИ ]**

* Сложность создания интуитивного гибридного редактора - смягчается использованием проверенных библиотек (CodeMirror, KaTeX, Bun) и фокусом на MVP с одним основным типом ввода.

* Попытка сделать «ещё один Moodle» - главный управленческий риск. Смягчение: жёсткое следование сформулированной философии «инструмент для преподавателя, а не система для организации».

* Недооценка необходимой простоты развертывания - смягчается фокусом на Docker и SQLite с первого этапа.
